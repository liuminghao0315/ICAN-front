🔍 分工概述

本分工基于《基于多模态深度学习的高校内容创作者行为分析平台》技术方案，将项目中的两位算法工程师和一位数据工程师的具体职责、任务目标和交付成果进行详细分解。三位成员的核心工作对应方案中的以下模块：

📹 视频处理模块

对应方案：技术架构-视频处理模块

负责人：项目负责人/CV算法工程师

🔊 音频分析模块

对应方案：技术架构-音频分析模块

负责人：算法工程师（音频/NLP）

📊 数据与评测模块

对应方案：工作流程-风险评估

负责人：数据工程师

项目负责人兼CV算法工程师

核心定位

负责视觉算法开发、多模态融合、整体架构设计与项目管理。重点对应方案中的视频处理模块和融合决策模块。

🎯 具体职责分解（基于方案）

1. 视频处理模块开发

对应方案章节：技术架构-视频处理模块、工作流程-多模态内容提取（视频模态）

  • 关键帧提取：实现每秒1帧的关键帧采样算法（方案工作流程2.1节）
  • 场景识别：开发校园场景分类模型，识别教室、图书馆、宿舍等环境
  • 人脸检测与表情分析：集成YOLO进行人脸检测，分析视频人物表情特征
  • 视觉特征提取：提取视觉特征向量，为多模态融合准备输入
技术栈： OpenCV PyTorch YOLO MoviePy
2. 多模态融合与风险评估

对应方案章节：技术架构-融合决策模块、工作流程-风险评估与预警

  • 三模态特征融合：设计视频+音频+文本的特征融合策略（方案2.4节）
  • 风险评估模型：实现综合风险评分算法，计算公式：
    综合风险 = 负面情感强度×0.3 + 高校声誉关联度×0.3 + 传播潜力×0.2 + 历史行为×0.2
  • 预警阈值调优：根据测试数据调整阈值（高风险>0.7，中风险>0.4，方案6节）
  • 高校视觉识别：检测校徽、校服等视觉标志（方案3.3节）
技术栈： PyTorch Scikit-learn ONNX Numpy
3. 架构设计与项目管理

对应方案章节：开发计划（8周冲刺）

  • 系统架构设计：定义模块接口规范，确保多模态协同工作
  • 进度管理：制定8周开发计划，监控各阶段里程碑
  • 技术协调：协调算法工程师和数据工程师的工作接口
  • 对外对接：与导师、需求方定期汇报项目进展

算法工程师（音频/NLP方向）

核心定位

负责音频处理和文本分析算法开发。重点对应方案中的音频分析模块和文本分析模块。

🎯 具体职责分解（基于方案）

1. 音频分析模块开发

对应方案章节：技术架构-音频分析模块、工作流程-多模态内容提取（音频模态）

  • 语音转文本：集成Whisper模型实现高精度语音识别（方案2.2节）
  • 声纹分析：识别说话人身份，判断是否为同一创作者
  • 音频情感分析：基于音调、语速、音量分析说话人情感状态
  • 音频特征提取：提取MFCC、韵律等音频特征
  • 高校关键词检测：从音频中识别高校名称、专业名称等
技术栈： Whisper Librosa PyAudio ONNX
2. 文本分析模块开发

对应方案章节：技术架构-文本分析模块、工作流程-舆论倾向预测

  • 文本预处理：使用Jieba分词，去除停用词，文本清洗
  • 情感分析：基于BERT-Chinese分析文本情感极性（正面/中性/负面）
  • 主题提取：识别视频主题（学习生活、校园恋爱、考试经验等，方案4节）
  • 关键词提取：提取高校相关关键词，构建高校知识图谱
  • 受众分析：基于文本内容预测受众年龄段和兴趣偏好（方案4节）
技术栈： BERT-Chinese Jieba SnowNLP Scikit-learn
3. 高校身份识别算法

对应方案章节：工作流程-高校身份识别

  • 文本高校识别：从标题、描述、字幕中提取高校信息
  • 音频高校识别：从语音内容识别高校表述
  • 高校知识库构建：维护985/211高校名称、简称、别称数据库
  • 身份声明检测：判断内容创作者是否明确声明高校身份
4. 模型优化与加速
  • 模型轻量化：优化音频和文本模型的推理速度
  • ONNX转换：将PyTorch模型转换为ONNX格式，提升部署效率
  • 性能测试：测试模型准确率，确保达到方案预期（高校身份识别>90%，舆论倾向预测>80%）

数据工程师（标注+评测）

核心定位

负责数据标注、模型评测和质量保障。贯穿整个方案，重点对应工作流程-风险评估和预期成果中的准确率指标。

🎯 具体职责分解（基于方案）

1. 数据标注体系建设

贯穿整个方案，支持所有模块

  • 标注平台搭建：部署Label-Studio，配置三模态标注界面
  • 标签体系设计：
    视频标签：
    • 场景类型（教室/图书馆/宿舍）
    • 人脸表情（高兴/中性/愤怒）
    • 校园标志（校徽/校服）
    音频标签：
    • 情感标签（积极/中性/消极）
    • 高校关键词
    • 声纹ID
    文本标签：
    • 情感极性（正/中/负）
    • 主题分类（6大类）
    • 高校关联度
  • 标注质量控制：制定标注规范，进行标注一致性检查
  • 数据集划分：划分训练集/验证集/测试集（比例70/15/15）
技术栈： Label-Studio Pandas SQL Python
2. 模型性能评测与分析

对应方案章节：预期成果-分析准确率

  • 评测指标设计：
    • 高校身份识别准确率（目标>90%）
    • 舆论倾向预测准确率（目标>80%）
    • 多模态融合效果评估
    • 各模块召回率、精确率、F1分数
  • 错误案例分析：
    • 每周分析模型预测错误的案例
    • 识别错误原因（数据不足/模型偏差/特征缺失）
    • 反馈给算法工程师进行模型优化
  • 性能监控：使用W&B跟踪模型训练指标，Prometheus监控线上性能
技术栈： W&B Prometheus Matplotlib Seaborn
3. 主动学习与数据迭代
  • 不确定性采样：选择模型预测不确定的样本进行标注
  • 数据分布优化：确保训练数据覆盖各类场景和主题
  • 反馈闭环：将用户反馈和错误案例纳入训练数据
  • 数据增强：对稀缺类别数据进行增强处理
4. 测试与质量保障
  • 测试数据集构建：构建覆盖各种场景的测试集
  • 端到端测试：模拟完整工作流程测试
  • 性能测试：测试系统并发处理能力
  • 回归测试：确保新版本不破坏原有功能

三位工程师的协作关系

工作流程协作关系图
阶段1：数据处理与标注

数据工程师主导：准备标注数据，搭建标注平台

算法工程师配合：提供标注需求和技术指导

阶段2：算法模块开发

并行开发：CV算法工程师开发视觉模块，音频/NLP算法工程师开发音频文本模块

数据工程师支持：提供训练数据，初步模型评测

阶段3：模块集成与融合

CV算法工程师主导：负责多模态融合和风险评估

算法工程师配合：提供音频/文本特征向量

数据工程师支持：评测融合模型效果

阶段4：模型优化迭代

数据工程师主导：分析错误案例，识别改进点

算法工程师执行：根据反馈优化各自模块

迭代循环：形成"开发→测试→反馈→优化"闭环

关键协作机制：
  • 每周算法评审会：两位算法工程师展示各自模块进展
  • 数据反馈会：数据工程师分享评测结果和错误案例
  • 联合调试：解决多模态融合中的接口问题
  • 标注规范共识：共同制定和更新标注标准
